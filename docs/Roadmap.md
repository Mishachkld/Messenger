# Roadmap: что делать на каждом этапе

Цели обучения: **React**, **SQL**, **WebSocket (WSS)**, **Redis**, **Docker**. Каждый этап даёт работающий результат — проект в любой момент можно показать.

**Стартовый уровень:** знаком C#/.NET; JS/TS, React и Go — с нуля. Поэтому перед этапами 2 и 6 заложены отдельные подготовительные шаги по языку — не пропускай их: React без понимания JS превращается в заклинания, которые «просто надо писать».

---

## Как получить максимум знаний (общие правила)

1. **Код пишешь сам, ИИ — ревьюер, а не автор.** Рабочий цикл: своя попытка → если застрял, вопрос «почему не работает / как это устроено», а не «напиши за меня». Сгенерированный код можно просить объяснить и критиковать твой, но вставлять его не глядя — значит потратить время проекта впустую.
2. **Сначала руками в консоли, потом в коде.** Каждую технологию щупай напрямую до того, как обернёшь в C#/TS: SQL — в `psql`/DBeaver, Redis — в `redis-cli`, HTTP — в `curl`, WebSocket — в `wscat` или DevTools. Так ты отделяешь «не понимаю технологию» от «не понимаю библиотеку».
3. **Ломай нарочно.** После каждой фичи устрой ей аварию: убей контейнер Postgres во время отправки сообщения, оборви Wi-Fi при открытом чате, отправь в WS мусорный JSON, удали индекс и посмотри на время запроса. Понимание системы = знание того, как она умирает.
4. **Веди журнал `docs/TIL.md`** (today I learned). После каждой сессии 2–5 строк: что нового узнал, что удивило, какой вопрос остался. Через полгода это самый ценный файл репозитория.
5. **Git-дисциплина.** Каждая фича — ветка; маленькие коммиты с осмысленными сообщениями; перед merge — прочитать собственный дифф целиком (self-review находит удивительно много).
6. **Читай первоисточники, а не пересказы**: доки PostgreSQL, react.dev, MDN (WebSocket), redis.io, Learn Go. Туториал с Medium — только чтобы понять, куда смотреть в доках.
7. **Тайм-бокс на затыки:** застрял больше чем на час — сформулируй вопрос письменно (часто это уже решает), запиши в TIL и либо спроси, либо отложи и иди дальше.
8. **В конце этапа — ретро:** объясни вслух (или письменно в TIL), как работает то, что построил, и почему именно так. Не можешь объяснить — этап не закончен.

## Этап 0. Инфраструктура (Docker)

**Учишься:** Docker, docker-compose, volumes, сети между контейнерами.

- [ ] `docker-compose.yml`: сервисы `postgres:16` и `redis:7`.
- [ ] Volume для данных Postgres (чтобы БД переживала перезапуск), проброс портов 5432/6379 наружу для отладки.
- [ ] Переменные окружения через `.env` (пароль БД и т.п.), `.env` — в `.gitignore`.
- [ ] Проверить подключение: `psql` (или DBeaver) к Postgres, `redis-cli ping` к Redis.
- [ ] Позже сюда добавятся `api`, `realtime`, `frontend` — каждый со своим `Dockerfile`.

**Готово, когда:** `docker compose up -d` поднимает БД и Redis, данные не теряются после `docker compose restart`.

**💡 Как выжать максимум:**
- Не копируй compose-файл целиком — пиши по строке и для каждой отвечай себе, зачем она (что такое healthcheck, чем volume отличается от bind mount, зачем контейнерам общая сеть).
- Залезь внутрь контейнеров: `docker exec -it <postgres> psql -U ...`, `docker exec -it <redis> redis-cli`. Пойми, что контейнер — это просто процесс с изолированной ФС.
- Эксперимент: удали volume (`docker compose down -v`) и убедись, что данные пропали. Теперь ты *знаешь*, а не веришь, зачем он нужен.
- Посмотри `docker compose logs -f`, `docker stats` — эти команды будут кормить тебя весь проект.

## Этап 1. API-скелет + auth (SQL начинается здесь)

**Учишься:** ASP.NET Core Minimal API, JWT, первые SQL-запросы (INSERT/SELECT), хеширование паролей.

- [ ] Проект `backend/api`. Подключение к Postgres через `Npgsql` + `Dapper`.
- [ ] Миграции как нумерованные `.sql`-файлы (`001_users.sql`, ...), раннер — DbUp или свой цикл на старте приложения. **Каждую таблицу пишешь руками через `CREATE TABLE`.**
- [ ] Таблица `users`. Эндпоинты: `POST /auth/register` (хеш пароля — `BCrypt.Net`), `POST /auth/login` → JWT (access ~15 мин + refresh).
- [ ] SQL, который здесь напишешь: `INSERT ... RETURNING id`, `SELECT ... WHERE username = @u`, уникальный индекс на `username` и обработка конфликта.
- [ ] Защищённый эндпоинт `GET /users/me` — проверить, что JWT-мидлварь работает.
- [ ] Dockerfile для API, добавить в compose (`depends_on: postgres` с healthcheck).

**Готово, когда:** через Swagger/curl можно зарегистрироваться, залогиниться и получить свой профиль по токену.

**💡 Как выжать максимум:**
- Каждый SQL-запрос сначала выполни в `psql`/DBeaver на живой БД и только потом переноси в код. Держи рядом файл `scratch.sql` с экспериментами.
- Разбери свой JWT на [jwt.io](https://jwt.io): посмотри header/payload/signature, поменяй один символ и убедись, что валидация падает. Пойми, почему токен нельзя «отозвать» и зачем нужен короткий срок жизни.
- Не бери шаблоны с готовым Identity/auth-скаффолдингом — вся ценность этапа в том, чтобы собрать auth руками и понять каждую строчку.
- Эксперимент: попробуй зарегистрировать двух пользователей с одним username параллельно (два curl подряд) — увидишь, зачем уникальный индекс, а не проверка `SELECT` перед `INSERT`.

## Этап 1.5. Основы JS/TS (перед React — обязательно)

**Учишься:** современный JavaScript и TypeScript как язык, без фреймворков. 1–2 недели.

JS выглядит знакомо после C#, но ведёт себя иначе — учить его «по ходу React» не выйдет, потому что будет непонятно, где кончается язык и начинается фреймворк.

- [ ] Пройти [learn.javascript.info](https://learn.javascript.info/) (есть русская версия), части 1 и нужное из части 2. Особое внимание темам, которых нет в C# или которые работают иначе:
  - замыкания и область видимости (`let`/`const`, что захватывает функция);
  - `this` (работает совсем не как в C#!) и стрелочные функции;
  - асинхронность: event loop, промисы, `async/await` — похоже на C# Task, но однопоточно;
  - деструктуризация, spread (`...`), методы массивов `map/filter/reduce` (это LINQ фронтенда);
  - равенство `===` vs `==`, truthy/falsy, `null` vs `undefined`;
  - модули `import/export`.
- [ ] TypeScript после этого «бесплатный»: типы как в C#, только структурные. Прочитать [TS Handbook «TS for C# programmers»](https://www.typescriptlang.org/docs/handbook/typescript-in-5-minutes-oop.html), понять `interface`, union-типы (`string | null`), дженерики.
- [ ] Практика без фреймворков: маленькая страница на чистом TS + Vite — форма, `fetch` к своему же API из этапа 1, вывод результата в DOM. Это «hello world» перед React.

**Готово, когда:** можешь без подсказок объяснить, что выведет код с замыканием в цикле и почему `this` внутри колбэка потерялся.

## Этап 2. React-фронт: логин и каркас

**Учишься:** React + TypeScript, роутинг, формы, работа с токенами, TanStack Query.

- [ ] `npm create vite@latest` → React + TS. Прописать proxy на API в `vite.config.ts` (чтобы не воевать с CORS в dev).
- [ ] Страницы: `/login`, `/register`, `/` (пока пустой список чатов). `react-router` + защищённые маршруты (нет токена → redirect на login).
- [ ] API-клиент: `fetch`-обёртка, access-токен в памяти/localStorage, авто-refresh по 401.
- [ ] TanStack Query для запросов (`useQuery` для `/users/me`), формы — обычный controlled state, без библиотек.

**Готово, когда:** можно зарегистрироваться и залогиниться из UI, после перезагрузки страницы сессия живёт.

**💡 Как выжать максимум:**
- Начни с официального туториала [react.dev/learn](https://react.dev/learn) (разделы Describing the UI, Adding Interactivity, Managing State) — он отличный и рассчитан на новичков в React.
- Держи в голове главную ментальную модель, непривычную после C#: **UI = функция от состояния**. Ты не «меняешь элементы» (как в WinForms/WPF), ты меняешь состояние, а React перерисовывает. Каждый раз, когда хочется «найти элемент и поменять» — это сигнал, что нужно новое состояние.
- Пока никаких UI-библиотек (MUI, antd) — обычный CSS или CSS-модули. Компонентная библиотека прячет от тебя и React, и CSS; красоту можно навести в конце.
- Прежде чем брать TanStack Query, один экран сделай на голом `useEffect + fetch` — с loading/error-состояниями и отменой запроса. Прочувствовав боль, ты поймёшь, *что именно* решает библиотека (и прочитаешь её доки осмысленно).
- Поставь React DevTools и смотри, какие компоненты перерисовываются и почему. Вопрос «почему этот компонент отрендерился 4 раза» — лучший учитель React.
- Разберись, где живёт токен и почему это компромисс (localStorage vs память vs httpOnly cookie) — запиши вывод в TIL.

## Этап 3. Чаты и сообщения через REST (основной SQL-этап)

**Учишься:** JOIN, GROUP BY, подзапросы, курсорная пагинация, индексы, транзакции. React: списки, формы, инвалидация кэша.

Бэкенд:
- [ ] Миграции: `chats`, `chat_members`, `messages` (+ индекс `(chat_id, id)`).
- [ ] `POST /chats` — личный чат с пользователем (в транзакции: chat + два member; проверка «личный чат уже существует» — самый интересный запрос этапа: self-join по `chat_members`).
- [ ] `GET /chats` — список чатов с последним сообщением и счётчиком непрочитанных. Это запрос с `JOIN` + `LEFT JOIN LATERAL` (последнее сообщение) + `COUNT` по `id > last_read_message_id`. **Написать его руками — половина пользы всего проекта.**
- [ ] `POST /chats/{id}/messages` (с `clientMessageId`, идемпотентность через `ON CONFLICT DO NOTHING`), `GET /chats/{id}/messages?before={id}&limit=50` — курсорная пагинация (не OFFSET!).
- [ ] `POST /chats/{id}/read` — `UPDATE chat_members SET last_read_message_id = ...`.
- [ ] Проверка прав в каждом запросе: пользователь — участник чата (`EXISTS`-подзапрос).
- [ ] Посмотреть `EXPLAIN ANALYZE` на список сообщений до и после добавления индекса — прочувствовать, зачем индексы.

Фронтенд:
- [ ] Список чатов, окно чата, отправка сообщений. Обновление — пока `refetchInterval: 3000` (polling). Оптимистичное добавление своего сообщения через `useMutation`.
- [ ] Поиск пользователей и создание чата.

**Готово, когда:** два пользователя в двух браузерах переписываются (с задержкой до 3 сек — это норм, real-time дальше).

**💡 Как выжать максимум:**
- Сгенерируй тестовые данные прямо SQL-ем: `INSERT ... SELECT ... FROM generate_series(1, 100000)` — 100k сообщений в чате. На маленькой таблице любой запрос быстрый, учиться не на чем.
- На этих 100k прогоняй `EXPLAIN (ANALYZE, BUFFERS)` для каждого запроса. Увидь своими глазами разницу Seq Scan vs Index Scan и почему пагинация через `OFFSET 90000` — катастрофа, а `WHERE id < @before` — мгновенна.
- Запрос «список чатов с последним сообщением и непрочитанными» напиши минимум двумя способами (LATERAL, оконная функция `ROW_NUMBER()`, подзапрос) и сравни планы. Это и есть обучение SQL.
- Устрой гонку: отправь два сообщения одновременно из двух вкладок, проверь порядок. Убей API посреди транзакции (создание чата) и убедись, что не осталось «половины» данных.
- На фронте сделай оптимистичное сообщение, а потом урони API и посмотри, что происходит с UI, — обработай откат мутации.

## Этап 4. Realtime-сервис: WSS + Redis Pub/Sub

**Учишься:** WebSocket-протокол руками (без SignalR), Redis Pub/Sub, межсервисное взаимодействие.

- [ ] Проект `backend/realtime` (ASP.NET Core, `System.Net.WebSockets`). Эндпоинт `/ws`, JWT — первым фреймом или query-параметром; валидация локально по общему ключу.
- [ ] In-memory реестр соединений: `ConcurrentDictionary<userId, список сокетов>` (несколько вкладок = несколько сокетов).
- [ ] API после сохранения сообщения публикует JSON-событие в Redis-канал `chat-events` (`StackExchange.Redis`). В событие включить список `memberIds` чата — realtime не должен ходить в БД.
- [ ] Realtime подписан на канал: получил событие → нашёл сокеты участников → разослал `{type: "message.new", payload}`.
- [ ] Ping/pong и закрытие мёртвых соединений по таймауту.
- [ ] Dockerfile, в compose. Проверить руками: `redis-cli` → `SUBSCRIBE chat-events` — видно проходящие события (отличный способ дебага).

Фронтенд:
- [ ] WS-модуль: connect, reconnect с экспоненциальным backoff, обработчики по `type`.
- [ ] `message.new` → дописать сообщение в кэш TanStack Query (`setQueryData`), убрать polling. Матчинг своих сообщений по `clientMessageId`.
- [ ] После reconnect — `GET /messages?after=<последний известный id>` для дозагрузки пропущенного.

**Готово, когда:** сообщения долетают мгновенно, обрыв сети (выключить Wi-Fi на 10 сек) переживается без потери сообщений.

**💡 Как выжать максимум:**
- Прежде чем писать сервис, подключись к чему-нибудь WebSocket-ному через `wscat` и посмотри фреймы во вкладке Network → WS в DevTools. Пойми handshake (`Upgrade: websocket`) — открой его в обычной вкладке Network.
- Напиши спецификацию своего протокола (типы сообщений, кто что шлёт) *до* кода — этот документ и есть контракт для будущей Go-версии. Любое изменение протокола — сначала в спеку.
- Держи открытым `redis-cli SUBSCRIBE chat-events` во время разработки — видеть события глазами полезнее любого логгера.
- Ломай соединения всеми способами: закрой ноутбук, убей вкладку, `docker restart realtime`, останови Redis. Для каждого случая ответь: потерялось ли сообщение и почему. Пойми главное: **Pub/Sub не хранит сообщения** — подписчик оффлайн = событие пропало. Поэтому истина всегда в Postgres, а дозагрузка после reconnect — обязательна.
- Открой 3 вкладки одним пользователем — все должны получать сообщения. Это проверяет, что реестр соединений «userId → список сокетов», а не «userId → сокет».

## Этап 5. Полировка realtime (Redis как хранилище)

**Учишься:** Redis не только как шина: TTL-ключи, паттерны presence. UX real-time-приложений.

- [ ] Typing: клиент шлёт `{type:"typing"}` в WS → realtime публикует в Redis → рассылка участникам. Троттлинг на клиенте (не чаще раза в 3 сек), индикатор гаснет сам через таймаут.
- [ ] Presence: при коннекте `SETEX presence:{userId} 60 "online"`, продление по ping-ам, при дисконнекте — удаление. Статусы в UI (зелёная точка).
- [ ] Read-receipts: `POST /read` → событие в Redis → у собеседника «прочитано».
- [ ] Непрочитанные в списке чатов обновляются по WS-событиям без refetch.

**💡 Как выжать максимум:**
- В `redis-cli` поиграй с TTL руками: `SETEX`, `TTL`, `KEYS presence:*` — посмотри, как ключ умирает сам. Подумай, почему presence через TTL надёжнее, чем событие «disconnect» (подсказка: процесс может умереть, не попрощавшись).
- Найди и почини гонку: пользователь с двумя вкладками закрывает одну — он всё ещё онлайн, но наивная реализация пометит его оффлайн.
- Троттлинг typing напиши сам (сравни throttle и debounce и выбери осознанно), а не бери lodash.
- Это хороший этап, чтобы впервые вернуться к старому коду и отрефакторить: WS-обработчики на фронте, наверное, уже превратились в switch-простыню — придумай структуру лучше.

## Этап 6. Переписывание realtime на Go

**Учишься:** Go: goroutines, каналы, `gorilla/websocket`/`nhooyr`, `go-redis`. Ценность контрактов между сервисами.

- [ ] Новый сервис на Go, реализующий **тот же протокол** и слушающий **тот же Redis-канал**. Спека уже есть — Architecture.md.
- [ ] Ключевые темы: goroutine на чтение + goroutine на запись для каждого соединения, реестр под `sync.RWMutex` (или через каналы), парсинг того же JWT.
- [ ] Поднять параллельно с .NET-версией на другом порту, фронт переключается переменной `VITE_WS_URL`. Погонять оба — поведение должно быть неотличимо.
- [ ] Выкинуть .NET-realtime из compose.

**Готово, когда:** фронтенд не знает, что сервер поменялся.

**💡 Как выжать максимум:**
- Go с нуля — это нормально: из всей тройки (JS, React, Go) он самый простой после C#. Сначала пройди [Tour of Go](https://go.dev/tour) целиком (день-два) и держи под рукой [Go by Example](https://gobyexample.com/) как справочник. Язык маленький — его реально выучить до проекта, а не по ходу.
- Перед самим сервисом напиши две разминки: CLI-утилиту (например, конвертер JSON) и эхо-сервер на WebSocket на 50 строк. После них основной сервис пойдёт легко.
- Главные непривычные после C# места: ошибки возвращаются значениями (`if err != nil`, никаких исключений), нет классов и наследования (структуры + интерфейсы, реализуемые неявно), goroutine — это не Task.
- **Не портируй .NET-код построчно.** Спроектируй по-гошному: две goroutine на соединение (reader/writer), fan-out через каналы. Дословный перевод с C# даст плохой Go и ничему не научит.
- Запусти с `go run -race` — детектор гонок найдёт ошибки в реестре соединений, которые ты иначе не увидишь.
- Финальный экзамен: нагрузочный мини-тест (скрипт, открывающий 1000 WS-соединений) против обеих версий — .NET и Go. Сравни память и CPU в `docker stats`, запиши выводы в TIL.

## Порядок и объём

Этапы строго последовательны, каждый — 1–3 недели неспешного темпа; этапы 2–3 с учётом изучения JS/React с нуля могут занять дольше — это нормально, скорость не цель. Самые «жирные» по обучению: **3** (SQL) и **4** (WSS + Redis). Не перескакивай через polling в этапе 3 — сделав сначала «плохо», лучше поймёшь, что именно даёт WebSocket.
